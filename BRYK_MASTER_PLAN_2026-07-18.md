## 0. Видение

Визуализатор для диджей-сетов на вечеринке. **Полностью отдельная аппка** (решение Роба 2026-07-18): не дека MUSICVIZ, не слой его стека — самостоятельный single-file инструмент, который берёт из наработок только проверенные модули.

Три кита:

1. **Геометрия как язык.** Прямоугольник, круг, треугольник — и всё, что из них собирается: сетки, кольца, колонны, туннели. Строгие формы, которые музыка заставляет жить.
2. **Цвет.** В отличие от ч/б канона флота — BRYK полноцветный: палитры-банк + цветовые оси, управляемые фичами звука. Это его собственная визуальная идентичность.
3. **Микрофон + камера.** Звук зала — через mic/line-in (audio-core). Камера — двумя ролями (решение Роба): **слой в миксе** (видео со своим blend/opacity под/над геометрией) и **текстура внутри фигур** (фигуры режут живое видео как маски).

Ключевой инсайт остаётся тем же, что в MUSICVIZ: ничего не строим с нуля — но здесь мы **копируем модули внутрь bryk/** (snapshot, правило одного владельца файла), а не шарим живые зависимости.

## 2. Архитектура

```
 bryk/
 ├─ index.html      ★ вся аппка (single-file: панель + движок + приёмы)
 ├─ audio-core.js   snapshot DSP (script src, same-origin)
 ├─ audio-core.test.js
 └─ serve.sh        localhost-сервер (getUserMedia не работает с file://)
```

### 2.1 Слои = стопка канвасов (не iframe)

Всё нативно, в одном DOM. Композит — CSS `mix-blend-mode` + `opacity` на стакнутых канвасах (GPU-композит, приём проверен в MUSICVIZ, но там были iframes — здесь дешевле):

```
 #stage
 ├─ #bg    фон: цвет/градиент из палитры (CSS или canvas)
 ├─ #cam   камера-слой: <canvas>, blend ▾ + opacity  ← роль «слой в миксе»
 ├─ #geo   геометрия: главный canvas, фигуры + camera-fill  ← роль «текстура в фигурах»
 └─ #fx    оверлей: grain/strobe/vignette/posterize (интенсивность ← фичи)
```

Порядок слоёв переключаем (камера над/под геометрией) — это z-index swap, бесплатно.

### 2.2 Камера-рельс (video-core, новый модуль)

- Пикер устройств: `enumerateDevices('videoinput')` + `getUserMedia({video:{deviceId}})` — та же грамматика, что audio-пикер MUSICVIZ (label-unlock после первого grant).
- Скрытый `<video autoplay muted playsinline>` → каждый кадр **один** `drawImage` в offscreen-снапшот → из снапшота уже и #cam-слой (cover-fit по shuffler-формуле), и заливки фигур. Один decode на кадр, сколько бы фигур ни резало видео.
- **Текстура в фигурах:** `ctx.save() → path(фигуры) → ctx.clip() → drawImage(snapshot, coverFit) → ctx.restore()`. До ~30–40 клипов на кадр — норм; массовые сетки заливаются видео через один общий clip составного пути.
- **Стилизация видео — обязательна:** клубная камера тёмная и шумная. Постеризация/duotone/threshold в цвета активной палитры превращает шум в фактуру и вписывает видео в цветовую систему (не «сырая вебка на экране»). Это дефолт, «чистое видео» — опция.
- Нет камеры / нет grant → слой честно выключен, фигуры фолбэк на палитровую заливку (никаких чёрных дыр).

### 2.3 Геометрия-движок

Сцена = **раскладка × материал × хореография** (три ортогональных оси — отсюда вариатива):

- **Примитивы:** rect, circle, triangle. Инстанс: `{shape, x, y, scale, rot, fill, stroke, order}`.
- **Раскладки (composition):** solo · grid N×M · ring (кольцо из N) · stack (концентрика) · columns · scatter (seeded) · tunnel (перспективная труба из уменьшающихся контуров) · mirror (зеркальные пары). Раскладка — чистая функция `(params, seed) → instances[]` (модель particle-dance).
- **Материалы (fill):** палитра-цвет · камера (clip) · контур (stroke only) · градиент палитры. Микс внутри раскладки: например каждая 3-я фигура — камера, остальные — цвет.
- **Хореография:** см. §4 «Приёмы». Позиция/масштаб/поворот/цвет каждого инстанса модулируются фичами.

### 2.4 Звук → движение

- `audio-core` как есть: `Audio.features = {rms, centroid, flux, onset, bands{6}, env{fast/slow}, beat{bpm, phase, beatN, barN, phraseN}}`.
- **Уровень 1 — зашитые связки в приёмах** (главный): каждый приём декларирует свои мишени (`pulse: scale ← env.bassFast` и т.п.). Диджею не нужна матрица — приёмы уже музыкальны.
- **Уровень 2 — мод-рэк для тонкой настройки** (фаза P3): портированная грамматика мод-строки MUSICVIZ поверх любого параметра активной сцены. Не строим раньше, чем заработает уровень 1.
- Квантование: смена сцен/приёмов — по `onBar`, переброс раскладки — по `onPhrase` (переключение не рвёт музыку).

### 2.5 Цветовая система

- **Палитра-банк:** 6–8 кураторских палитр по 4–5 цветов (bg + 3–4 фигурных). Палитра — юнит переключения (клавиши/onBar), не отдельные пипетки.
- **Цветовые оси от звука:** hue-drift ← centroid (яркость спектра = теплота), lightness-пульс ← env.bassFast, color-flip (перестановка ролей внутри палитры) ← onset/bar.
- Интерполяция в OKLCH (перелив без грязи), хранение — токены палитры.
- Duotone камеры — из активной палитры (§2.2), чтобы видео и геометрия жили в одном цвете.

### 2.6 Перформанс-режим

- `ARM AUDIO` (жест обязателен для AudioContext) → пикеры mic/cam → метры.
- Клавиатура: `1–8` сцены · `T` tap ×4 · `[`/`]` nudge · `F` fullscreen · `H` скрыть весь хром · `B` blackout · `C` камера on/off · `P` следующая палитра · `Space` следующий приём (quantized).
- Сцен-банк 8 слотов, localStorage + export/import json (модель MUSICVIZ v4: input, слои, раскладка, палитра, приём, rows).
- Вывод: fullscreen на проектор/ТВ, foreground-вкладка (фоновая паузит rAF).

## 3. Почему не WebGL (решение по стеку)

Canvas 2D: примитивы-контуры, clip-заливки видео, blend через CSS — всё в бюджете 60fps при десятках фигур (particle-dance гоняет сотни точек в 2D без проблем; drawImage видео 1×/кадр — копейки). WebGL дал бы тысячи инстансов и шейдерные FX, но стоит недели отладки (см. Particles-3D blank-render сагу). Правило: **2D до упора, WebGL только если конкретный приём в него не влез** — тогда это отдельное решение с отдельной строкой в этом доке.

## 4. Приёмы (каталог вариативы, v1)

Хореография, сгруппированная по музыкальной драматургии. Каждый приём = декларация связок + кривая.

**На удар (fast):**
- **Pulse** — scale фигур ← env.bassFast (exp-кривая, разный depth по раскладке).
- **Strobe-flip** — onset: инверсия fill↔stroke или color-flip палитры, decay 120ms.
- **Kick-scatter** — onset: разлёт инстансов от центра с пружинным возвратом.

**На бит/такт (clock):**
- **Spin** — rot ← beat.phase (кольца и туннели — гипноз на ровной фазе).
- **Shuffle** — onBar: seeded-перестановка фигур по слотам раскладки (paint-order по урокам shuffler: входящая ПОД, центральная сверху).
- **Cascade** — волна scale/цвета по инстансам с лагом i·60ms от удара.

**На дыхание (slow):**
- **Breathe** — scale/интерлиньяж сетки ← env.rmsSlow.
- **Tunnel-zoom** — скорость движения по трубе ← env.bassSlow.
- **Hue-drift** — медленный увод палитры ← centroid.

**Драматургия (build-up/drop):**
- **Собраться** — flux растёт (build-up) → раскладка расшатывается в хаос; drop (onset после пика) → мгновенный снап в строгую сетку. Перенос guilloché chaos→order на геометрию — главный wow-приём.
- **Врубить толпу** — на drop камера-слой вспыхивает в полную opacity/screen на 2 такта → уходит.

Вариатива = раскладки (8) × материалы (4) × приёмы (11) — сотни состояний из малого словаря. Новые приёмы дописывать сюда датированно.

### 7.2 Ключевой ход: ВСЁ — текстура (один пайплайн на три источника)

Движок рисует текстурные квады — значит, geometry/PNG/SVG/камера унифицируются на входе:

1. **PNG** — drop, как в shuffler (`addFiles` 679).
2. **SVG** — drop → `Image` → растр в texture-canvas (тот же `buildTexture`); перерастеризация при смене размера — дёшево.
3. **Примитивы rect/circle/triangle** — *процедурные текстуры*: фигура рисуется в offscreen с цветами активной палитры (fill/stroke/градиент). Смена палитры = перерисовка текстур, не сцены.
4. **Камера** — video-снапшот (§2.2) → texture-canvas, обновляется каждый кадр. **Упрощение против §2.2:** маска формы (круг/треугольник режет видео) делается clip'ом ОДИН раз внутри texture-canvas, а не на каждый инстанс сцены — сколько угодно летающих «видео-кругов» за один clip. Duotone-стилизация — там же, на снапшоте.

Итого фигура BRYK = текстура × item-контракт. «Летающий треугольник с толпой внутри, поворачивающийся ребром» — это texture(камера+маска) + `oy` анимация. Ничего нового в рендере.

### 7.3 Добавка из particle-dance (`particle-dance-builder/index.html`)

`patternPoint(p, pattern, t, opts)` — 15 seeded-паттернов (cloud · lattice · vortex · ribbon · burst · wave · spring · zigzag · spiral/DNA · axes · orbitals · fan · steps · glyphs · form) с `spread/dance/amplitude/frequency/twist/lanes`, orientMode и depthFade. Модель совместимая: чистая функция → позиции. **Адаптер в item-контракт (билборды малого размера) = второй банк раскладок** — рой мелких фигур против крупных квадов shuffler'а. Это и есть два регистра подачи: «карточки летят» (shuffler) и «масса танцует» (particle-dance) в одном движке.

### 8.2 Архитектура хореографии — три ортогональных регистра (финальная модель)

```
 цель (target-провайдер)      ×   транспорт к цели         ×   локальные взаимодействия
 ─ LAYOUTS shuffler (§7.1)        ─ lerp-follow (как есть)     ─ collide / push / avoid / spin
 ─ patternPoint рой (§7.3)        ─ springAccel (пружина)      ─ boids (стая)
 ─ Formation-маска (8.1)          ─ силы: attract/curl/gravity ─ crowd-глушение
```

Физика работает в world-space ПОВЕРХ item-контракта §7.1: силы двигают `it.x/y` (velocity-интеграция) к целям раскладки, z остаётся от раскладки, рендер не меняется. Никакой 3D-физики не строим — плоскость сил + 2.5D подача, этого достаточно и это честно по FPS.

### 8.3 Прокачка: новые приёмы на физике (дополнение каталога §4)

- **«Магнит-дроп»** — build-up: `attract` всасывает всё к центру, urge ← flux; на drop — `Scatter`-импульс наружу + `persp` в пик. Драматургия сжатие→взрыв.

- **«Гравитация»** — брейкдаун: Fall — всё падает и пилится у пола (resolveBodies); возврат бита — гравитация инвертируется/выключается, всплытие к раскладке.
- **«Слово»** ★ — Formation-маска: фигуры собираются в текст (BRYK · имя трека · DROP) или в SVG-лого (маска из ассета — тот же drop-пайплайн §7.2) на onPhrase; Fill/Frame режимы. Текст задаётся с клавиатуры прямо в сете.

- **«Расталкивание»** — плотный grid + collide: onset → импульс случайному инстансу → цепная реакция по соседям.
- **Пружинные переходы** — смена раскладок через springAccel вместо снапа: стиль прилёта задаёт Motion Pad (Y-ось = характер пружины).

## 9. 2026-07-18 · Единая сцена: сведение режимов, мастеринг, типографика

### 9.0 Скан истории — уроки EQJ

Стек EQJ/MUSICVIZ — это уже была попытка совмещения именно этих четырёх движков: `mvj-inject.html:322` — **LINES** (Synthex) · **BODIES** (verlet) · **DOTS** (particle-dance) · **FORMS** (Motion Primer), слои-iframe + нативный, комб-микшер v3 (строка: n·имя·opacity·M·S·blend), карточки маппинга. 2. **Слои не видят друг друга:** композит только blend/opacity картинок — режимы не взаимодействуют. → BRYK: единый пул (9.1).

### 9.1 Один скрин = один пул (главное решение)

НЕ стопка независимых слоёв-картинок. Один canvas #geo, **один пул тел** `{x,y,z,vx,vy,texture,face}`, а режимы — «программы» над пулом:

```
режим (mode) = target-провайдер (§8.2) + набор сил + материал(текстуры)
         каждый активный режим имеет ВЕС ПРИСУТСТВИЯ w ∈ 0..1
```

- **Сведение = веса, не прозрачность.** Цель тела = взвешенный бленд целей активных режимов (веса нормируются); силы аддитивны ×w; материал кроссфейдится. `w→0` — режим *тает из процесса* (перестаёт влиять на тела), а не «выцветает картинкой». Это формализация Робова «убирали присутствие значения эффекта — и легче сводили».
- **Морф A→B = частный случай:** два режима, один фейдер (`w_B = x, w_A = 1−x`). Переход квантуется onBar/onPhrase, кривая шага — Motion Pad. Степ-переходы между режимами — это он.
- **Взаимодействие режимов:** collide/boids/crowd (§8.1) работают на ВСЁМ пуле независимо от того, чья цель у тела — тела разных режимов физически замечают друг друга. Два способа делить пул (оба нужны, по приёму): **blend** — все тела слушают все режимы через веса; **split** — доли пула закреплены за режимами (рой DOTS расталкивается с квадами карусели).

### 9.2 Мастеринг (вместо покруток каждого слоя)

- **Карточка режима:** имя · большой фейдер **PRESENCE** (w) · M/S · disclosure с параметрами внутри. Снаружи — один регулятор.
- **Мастер-шина: 3 макро** — `ENERGY` · `DENSITY` · `CHAOS`. Каждый макрос разложен на параметры активных режимов декларацией в пресете (макро-маппинг). DJ вживую рулит: presence-фейдеры + 3 макро + сцены. Всё остальное — подготовка до сета.
- Банк сцен сохраняет: состав режимов + веса + макро-раскладку + палитру (эволюция сцен v4 MUSICVIZ).

### 9.3 UI одного скрина

Левая **нав-панель** (разделы: MIX · MODES · AUDIO · LOOK · SCENES) + колонка **карточек-слоёв** (стек карточек активных режимов, грамматика строки из комб-микшера v3: n·имя·PRESENCE·M/S) — сцена занимает остальной экран. ### 9.4 Формация + типографика (уточнение к 8.3 «Слово»)

### 10.4 Модуль маппинга — `music-viz/mvj-inject.html` (144КБ, 13.07)

**Терминология:** «карточка маппинга» = **мод-строка**, DOM-ряд. Не картинка, не ассет. Сущности «card» в коде нет.

**Модель — три уровня:**
```
layers[]  (layers[0] = ВЕРХНИЙ)
  └─ L = {id, engine, label, blend, opacity, muted, solo, matrix:[], ifParams:[], ifMap:{}, …}   (:670)
       └─ matrix[i] = row   ← одна мод-строка
```
Глобальный `matrix` — **ссылка**, не копия, на `activeLayer().matrix` (:549, переприсваивается :654, :678–683, :919).

**Строка** (`defRowFor` :839–841): `{feat, path, mode, base, depth, curve, atk, rel, gate, nu, lo, hi, mute, solo}` + рантайм `_v, _c, _ns, _lrn, _lo, _hi, _last, _els, _scrubs`. `ROW_KEYS` (:881) — 14 persist-полей. **`nu` (noise underlay) — восьмой элемент грамматики, в §2.4 плана его не было.**

**Ядро — `processMat` (:964–983).** Конвейер: `raw → нормализация lo/hi → curve → gate (S&H) → noise-mix → mode/depth вокруг base → clamp → attack/release → apply`.

Умные детали: `env.*` **пропускают** свой atk/rel (`if(r.feat.indexOf('env.')===0) r._v=target`) — audio-core их уже сгладил, второй фильтр даёт только лаг · при смене `path` строка перезабирает base и, **если параметр тяжёлый (`p.rebuild`), автоматически ставит `gate='beat'`** (:857) · gate это **sample-and-hold**, а не ворота в ноль (:971) — значение кривой замораживается в `_c` и обновляется только на импульс, поэтому можно модулировать дорогие rebuild-параметры.

🔴 **Конфликт строк: last-wins, не сумма.** `applyOne` (:564) — простой сеттер без аккумуляции, цикл идёт по массиву, каждая строка пишет поверх предыдущей. Две строки на один параметр → работает **только последняя**, первая жжёт CPU. **Для BRYK с весами присутствия это блокер.** Патч — аккумулятор отклонений от базы:
```js
var acc=Object.create(null);
// в теле цикла вместо applyOne:
var slot=acc[r.path]||(acc[r.path]={p:p,sum:0,n:0,base:r.base});
if(r.mode==='abs'){ slot.sum=r._v; slot.abs=true; }      // abs — override
else { slot.sum+=(r._v-r.base); slot.n++; }              // остальные — сумма ОТКЛОНЕНИЙ
// после цикла:
for(var k in acc){ var s=acc[k]; applyOne(kind,s.p, s.abs?s.sum:clamp(s.base+s.sum,s.p.min,s.p.max)); }
```

**Контракт `Param` — самое ценное для переноса.** Три провайдера, один контракт:
```js
{ key, label, min, max, step, rebuild,        // rebuild:true → троттлинг + авто-gate на бит
  scope:'src'|'root' | get()/set(v) | el }    // ОДИН из трёх способов доступа
```
- **(A) курируемый каталог** `SP` (:301–316) + dot-path walker `sBase/sGet/sSet`. **Не авто-обход state** — ручной список. Умышленно: :309 объясняет, что `post.*` выпилен не по лени, а потому что дорог и плохо выглядит. **Каталог = художественное решение о том, что стоит модулировать.**

**Фичи** (`FEATURES` :699, 19 шт; `featVal` :703): `rms · flux (×8, мал по природе) · onset · band.{sub,bass,lowmid,mid,highmid,treble} · env.{bassFast,bassSlow,midFast,midSlow,highFast,highSlow,rmsSlow} · beat.phase · beat.sine · beat.pulse`. **Три `beat.*` — производные, считаются в хосте, не в audio-core:** `sine = 0.5-0.5*cos(2π·phase)`, `pulse = pow(max(0,1-phase),3)`.

### 10.6 Цвет и шрифт — что есть и чего нет

**Пикера-окна в Synthex НЕТ.** «Окно с ползунками и пипеткой» — это **нативный `<input type="color">`**, окно рисует macOS (её системная панель действительно имеет и ползунки, и экранную пипетку). Своей HSL/OKLCH-плоскости, hue-кольца и пипетки в движке не написано. Для BRYK это значит: механика бесплатна и работает, но **окно нельзя стилизовать** — на тёмной аппке откроется macOS-панель, не наша.

**Банк цветов ЕСТЬ** — стопы градиента, `buildStops()` + `#addStop` (:5327–5352), CSS `.crow.stoprow` (:279–283). Ряд = свотч `.spick` + hex-поле + позиция + `×`. **Ёмкость 2–6 жёстко — для BRYK поднять (Роб: 8 и больше).** Три решения забрать дословно:
1. `+ Color` ищет **самый широкий промежуток** и ставит стоп в его середину, не в конец списка.
2. Цвет нового стопа берётся **из уже посчитанной рампы в точке p** — добавление цвета не меняет сам градиент, только даёт новую ручку.
3. `del` прячется через `visibility`, не `display` — сетка ряда не прыгает при 3→2.

`normStops()` (:1870–1876) — единая точка санации: чинит битый массив, клампит позиции, валидирует hex, **сортирует по p**. Переупорядочивания драгом нет и не нужно: порядок — производная позиции.

**Полоса-рампа `#rampBar`** (:5220–5255, :5356–5385) — контрол с двумя смыслами на одной поверхности: драг ручки = позиция стопа, драг между ручками = сила рампы, dblclick = добавить стоп. Приём `amtHist`: dblclick прилетает **после** двух `pointerdown`, каждый уже подвинул `amount` — история из двух значений откатывает сдвиг.

**Две цветовые структуры, разного рода — не путать:**

**(A) `state.palette` — НЕ банк.** Ровно 4 слота фиксированной длины, добавления/удаления нет (:1332): `[0]=A/левый край`, `[1]=Paper`, `[2]=B/правый край`, `[3]=Lead`. Свои пикеры только у `[1]` и `[3]`. **У A и B пикеров нет намеренно: `syncRamps()` записывает края обратно в `palette[0]`/`palette[2]` (:1878)** — один источник правды, это стопы. Санация битого массива при загрузке cfg :1405–1406 (`palette[1]=undefined` роняло `paperNow→invHex→hexToRgb`).

**(B) `state.tone.stops` — настоящий банк.** `[{c:'#rrggbb', p:0..1}]`, 2–6 штук, с добавлением и удалением.

**Хранение — плоские hex-строки, но смешивание УЖЕ в OKLab.** Это поправка к §2.5: перцептивная интерполяция строить не надо, она есть. `mixHex(a,b,t)` через `srgbToOklab` (:713) и `makeRamp(stops,n)` (:716–719) — чистая самодостаточная математика, конвертит стопы в OKLab и сортит по pos. Формат хранения при этом остаётся hex, поэтому сериализация в cfg тривиальна (`JSON.stringify`, :1804). **CSS-переменных в цветовом пути нет вообще** — токены `:root` красят только хром панели; цвет артефакта идёт прямым чтением state в canvas-код: `syncRamps()` печёт **LUT на 256 позиций** `toneRamp.css[0..255]`, рендер бьёт индексом (:1922, :2956, :3161, :3376, :5246). `TINT` (:1986) — Map-кэш производных оттенков, сбрасывается в `syncRamps`.

**Извлекаемый блок цвета: ~130 строк JS + ~12 CSS**, зависимости только `clamp01`/`clamp` и токены. `normStops` (1870–1876) · `syncRamps` (1877–1880) · `makeRamp/mixHex/hexToRgb/srgbToOklab` (709–719, самодостаточны) · `buildStops`+`#addStop` (5327–5352) · `RB`/`rbX`/`rbP`/`drawRampBar` (5227–5255) · ramp-bar жесты (5356–5385) · `toHex` (5217–5219) · `syncPalette`/`applyColour` (5388–5396) · CSS (118–129, 279–283). `stepT` (:1903–1907) нужен только под ступенчатую рампу, иначе `t=>t`; `viewHex` заглушить как `h=>h`, если инверсия не переносится.

⚠ Баг задокументирован в CSS (:118–126) и стоит перенести вместе с кодом: `.cpick` (роль, есть `data-i`) и `.spick` (стоп, `data-i` нет) **разведены по классам** — раньше оба были `.cpick`, `syncPalette` шла по всем, у стопа `+undefined = NaN`, `palette[NaN] = undefined`, `toHex(undefined) = #000000`, и любая правка Lead красила все свотчи стопов в чёрный. **Один класс — один смысл.**

**Выбора шрифта НЕТ вообще.** Ноль по `queryLocalFonts` / `new FontFace` / `document.fonts.*` / шрифтовой UI / drag-drop `.ttf` — по всей папке `Projects`. ### 11.5 Новое: честный спектр

🔴 **Найденное:** «распределения частот» в EQJ по сути не было. Шесть полос `sub…treble` — единственное, что видел пользователь, и **каждая нормализована своим скользящим AGC** (`audio-core.js:52–54`). Поэтому их высоты **между собой несопоставимы**: `sub` в тишине показывает то же, что `bass` на бочке. Как индикатор «полоса живёт» работает, как картина спектра вводит в заблуждение.

Добавлен канвас спектра из **сырых бинов** по логарифмической оси частот 20 Гц … 16 кГц, с медленным peak-hold и рисками границ шести полос, чтобы метры ниже можно было найти в картине. Бренд-градиент синий → жёлтый → красный по оси частот. Подпись под ним честно говорит, что полосы ниже нормализованы независимо.

### 11.6 Ревью `apple-design` — что применено

1. **Резинка вместо порога (находка №1 из §10.9 — закрыта).** Скраб Synthex держит жёсткие 28px и отпускает. Apple требует прогрессивного сопротивления: жёсткий порог читается как «замёрзло», нарастающее — как «отзывается, но дальше ничего нет». Чистая асимптотическая резинка тоже не годится, она бы не пустила дальше, а требование Роба (W-ROB-20) — доехать куда угодно. Решение: резинка на первом участке, затем плавный возврат к 1:1. Непрерывно и по значению, и по производной.
   ```js
   const RESIST_PX = 26;
   function resist(over){ const c=0.55;
     const soft=(over*RESIST_PX*c)/(RESIST_PX+c*over);   // асимптота (Apple rubberband)
     const t=Math.min(1, over/(RESIST_PX*2));            // 0→1 на двойной зоне
     return soft*(1-t) + over*t; }                       // …и отпускаем в линейность
   ```
2. **Отклик на pointer-down**, 3px гистерезис до перехода в drag, `scale(.97)` на нажатии — только transform.
3. **Лампа бита с пружинным оседанием** вместо линейного спада: атака 0.55, релиз 0.12. Удар читается телом, а не графиком.
4. **Пространственная привязка**: тулбар угла растёт из самой кнопки (`transform-origin:100% 100%`), а не из центра. Вход и выход по одному пути.
5. **Из ловушки есть выход**: при скрытом хроме угловая кнопка остаётся (гаснет до 18%, поднимается на hover/focus). Мёртвый угол оставил бы пользователя без пути назад.
6. **`backdrop-filter` на панели не ставим** — панель висит над канвасом на 60fps, блюр стоит кадров. Только после замера.
7. **reduced-motion** гасит всю панельную анимацию, `prefers-contrast:more` усиливает разделители.

### 11.7 Найденный баг канона Synthex (ретрофит)

🔴 **Легенды групп клипаются под пользовательским переопределением интервалов (WCAG 1.4.12).** В движке `legend` позиционирована абсолютно с **фиксированной `height:14px`** (`engine:206–218`). Фиксированная высота на текстовом контейнере — прямое нарушение закона §5 `adaptive-typography`; при включённом стресс-тесте текст обрезается.

Абсолютная посадка была нужна, чтобы нативная рамка `fieldset` не шла сквозь текст. Но у нас (и у движка) `border:0`, разделитель рисуется `box-shadow: inset 0 -1px`. Рамки нет — трюк не нужен. В BRYK легенда переведена в обычный поток без фиксации высоты.

**Это стоит отнести обратно в Synthex Engine** — там ровно та же конструкция и, вероятно, ровно тот же дефект.

- **Единый пул** (§9.1): один массив тел, режимы — «программы» над ним. Реализованы пять: `grid · ring · tunnel · scatter · word`.

### 12.2 🔴 Найденное и починенное: бюджет квадов

Перспективный квад стоит клип + аффинный `drawImage` на ячейку сетки, и в Canvas2D это дорого независимо от плотности сетки. Замерено на старте: **300 развёрнутых тел = 20 fps**, при огрублении сетки до 2×2 — всего 26.

Причина архитектурная: движок шаффлера строился под пару десятков крупных кадров-фотографий, а не под рой. §10.3 это предсказывал; цифра оказалась хуже ожидаемой.

Два хода, второй решающий:
1. **LOD** — тело мельче 34px рисуется билбордом (своей перспективы оно всё равно не покажет), 34–90px получает сетку 2×2 вместо 4×4. Дало 20 → 26 fps. Мало.
2. **Бюджет квадов** — полную обработку получают только **80 самых крупных** тел кадра, остальные билбордятся. Порог вычисляется по `P.k` каждый кадр. **Цена кадра перестала зависеть от количества тел.**

Итог (headed, реальное окно): 60 fps на всех сценах, включая **800 тел с тремя режимами в блендинге и аберрацией**. До правки эта же сцена не поднималась выше 20.

Правило на будущее: **любой новый эффект per-body с клипом обязан иметь бюджет.** Без него сцена деградирует не постепенно, а обвалом.

### 12.3 🔴 FPS в headless не гейтится — и не должен

Headless Chromium рисует канвас через SwiftShader без vsync. На одном и том же коде: 300 билбордов — 110 fps, 300 развёрнутых — 31 fps; в headed-окне обе сцены дают ровные 60. Цифры не про машину Роба и не сравнимы с ней.

`probe/app.cjs` печатает fps **информационно** и гейтит вместо него **форму кривой**: билбордный путь масштабируется сублинейно, а бюджет квадов держит развёрнутый путь (1000 развёрнутых ≥ 60% от 300). Это то же решение, к которому пришёл Synthex Engine («perf-line — INFORMATIONAL, never gates — headed-only»).

Судить кадры — в реальном окне, на боевом ноуте.

### 13.3 Физика — третий регистр (§8.2 наконец собран)

Силы двигают тело в world-space **поверх** целей режимов, поэтому режим остаётся провайдером цели и ничего не знает про столкновения. При `Physics on` транспорт к цели превращается из lerp в **пружину на той же скорости**, что и силы — отсюда «тело тянет раскладка и толкает сосед в одном кадре».

| Сила | Как сделано |
|---|---|
| **Collide** | попарное расталкивание через **равномерный пространственный хэш**. План (§8.5) закладывал потолок ~150 тел на O(N²) — с хэшем те же силы держат 900+, потому что цена идёт по плотности, а не по популяции |
| **Swirl** | curl-поле аналитически из двух синусов: `ψ = sin(a)·cos(b)`, ротор берётся руками. Дивергенция нулевая по построению, поэтому тела **закручиваются, а не стекаются в сток**. Ноль зависимостей, не тянем реализацию шума |
| **Attract** | магнит с усилением у центра (`boost/(d+soft)`, Motion Primer) |
| **Gravity** | биполярная, вверх тоже |
| **Flock** | когезия к локальному среднему, переиспользует ту же решётку |
| **Damping / Follow** | характер прилёта |

🔴 **Найдено при проверке: магнит был слабее раскладки.** При `attract=2` и `follow=3` рой оставался на радиусе 2.4 — пружина к сетке била магнит в пять раз. Это ломало приём «Магнит-дроп» (§8.3) в принципе: build-up обязан уметь всосать всё к центру. Коэффициент поднят так, чтобы магнит выигрывал на максимуме. Проверено: 2.78 → **0.73** при включении, обратно к 2.78 при отпускании, без схлопывания в точку.

## 14. 2026-07-18 · Камера-рельс — третий кит из §0 закрыт

### 14.1 Обе роли камеры реализованы

Как и решено в §0.3 — камера входит **двумя ролями**, и обе живут на одном декоде:

1. **Текстура внутри фигур.** Фигуры режут живое видео. Реализовано по §7.2.4: маска формы применяется **один раз внутри texture-canvas**, а не на каждый инстанс. Три canvas'а на кадр (rect/circle/tri) обслуживают сколько угодно летающих «видео-кругов». Проверено: центр текстуры непрозрачен, угол круглой текстуры пуст — маска реально режет.
2. **Слой в миксе.** Отдельный `#camcv` под или над геометрией, со своим blend (screen/normal/multiply/overlay/difference/lighten) и прозрачностью. Порядок слоёв — переключение z-index, бесплатно.

Камера входит в тот же **микс форм**, что примитивы и ассеты: `camW` это ещё один вес. «Летающий треугольник с толпой внутри» — просто запись в той же раздаче.

### 14.2 Стилизация — не опция, а дефолт

§2.2 требовал этого прямо, и требование верное: клубная камера тёмная и шумная, сырая вебка на проекторе читается как ошибка. Конвейер: `contrast/brightness/saturate` фильтром → **дуотон в активную палитру** (multiply тёмным стопом + screen светлым, два прохода держат средние тона читаемыми). Ползунки Duotone / Contrast / Gain, зеркало флагом. «Чистое видео» — это дуотон в ноль, то есть опция.

Дуотон уже положил видео в палитру, поэтому камера-текстуры **не размножаются по цветам палитры** — перекраска по стопам только замутила бы их. Три текстуры, не девять.

### 14.3 Честный фолбэк

Нет камеры или отказ в правах → слой выключен, камера-лица не входят в микс вообще, фигуры остаются на палитровых заливках. Никаких чёрных дыр (требование §2.2). Сообщение человеческое, кнопка уходит в RETRY. Клавиша `C` включает и выключает.

### 14.4 🔴 Найденный баг специфичности CSS

`#camcv{display:none}` **никогда не применялось**: селектор `#stage canvas` это id + элемент (0,1,1) и он бьёт голый id (0,1,0). Тот же дефект был у `#glcv` и его маскировал инлайн-стиль, который пишет кадровый цикл — то есть пост-слой скрывался не правилом, а побочным эффектом рендера. При загрузке до первого кадра оба оверлея были `block`.

Исправлено поднятием специфичности (`#stage #glcv`, `#stage #camcv`). **Урок: если правило работает только потому, что JS каждый кадр пишет инлайн-стиль, это не правило.**

### 14.5 Как проверялась камера без камеры

Гейт не может рассчитывать на физическое устройство. Пайплайн проверяется подстановкой: `videoWidth/videoHeight/readyState` подменяются на видео-элементе, а `drawImage` перехватывается и подставляет синтетический кадр вместо `<video>`. ### 17.1 Логика панели — кто за что отвечает (разжёвано)

Принцип: у каждого параметра ЧЕТЫРЕ возможных автора — рука · приём · рэк · сцена. Все пишут через один вход (`setDriven`/базы, §15.5). Панель делится не по «где влезло», а по ВРЕМЕНИ использования:

```
 ЛЕВАЯ ПАНЕЛЬ = ЧТО НА СЦЕНЕ (глаза)          ПРАВАЯ ПАНЕЛЬ = МУЗЫКА (уши)
 ┌────────────────────────────────┐            ┌────────────────────────────────┐
 │ MACRO   ENERGY·DENSITY·CHAOS   │ ← сет      │ INPUT   источник·ARM·CAL·метры │ ← настройка
 │ MODES   карточки: PRESENCE+M/S │ ← сет      │ CLOCK   BPM·tap·trim·фаза     │ ← сет (tap)
 │   Pattern A ▸ 17 · Pattern B ▸ │ ← сет      │ RACK    мод-строки (схлопнут) │ ← подготовка
 │ LOOK    палитра·пост-слот      │ ← подгот.  │ SCENES  банк 8·export/import  │ ← сет (вызов)
 │ WORD    текст·шрифт·Fill/Frame │ ← сет      └────────────────────────────────┘
 └────────────────────────────────┘
 ХОТКЕИ = ПЕРФОРМАНС: приёмы 1–4 · сцены 5–8 · QUANT · T tap · B blackout · H хром · F fullscreen
```

- **В сете** DJ трогает: 3 MACRO, PRESENCE-фейдеры, морф A/B, WORD-текст, tap, вызов сцен, приёмы. Всё остальное — подготовка дома.
- **MACRO — верх левой** (агрегат «что на сцене»); раскладка макро→параметры декларируется в пресете и живёт в сцене. ENERGY→speed/attract/kick-глубины · DENSITY→count/gap/scale · CHAOS→jitter/swirl/manner-сдвиг.
- **Иерархия авторов при конфликте:** рука > приём > рэк (рука перебазирует; приём возвращает заимствованное в done(); рэк модулирует вокруг базы). Сцена = смена всех баз разом.

### 17.2 Практики «слышать музыку на тусе» (ресёрч по статьям/книгам)

**Вход — иерархия честности** (единодушно во всех источниках): 1) booth/line-out с DJ-микшера (или сплиттер) — профессиональный стандарт; 2) BlackHole для системного звука; 3) микрофон — фолбэк, и тогда направленный, у динамика, не в толпе.

**Шумоглушение при mic-входе:**
- **CAL-кнопка** (в INPUT): захват 5–10с фона площадки → noise-floor по полосам → спектральное вычитание/порог. Разовая калибровка на месте.
- **Бит слушать в sub/bass** — толпа шумит в mid/high, кик пробивает низом; highpass <25Гц против гула вентиляции. Полосный селектор «что считать битом».
- **Onset-детектор**: refractory-период ~100мс (двойных срабатываний нет), гистерезис порога, adaptive whitening (Stowell & Plumbley 2007) — выравнивание спектра до детекта, классика для шумных сцен. Наш median+MAD уже хорош, whitening — апгрейд.
- BPM: confidence-гейт (авто ниже порога → держать последний), **tap всегда истина** (уже канон).

**Как часто реагировать (пейсинг):** change-бюджет по уровням — мелкое (пульс/цвет) на бит · среднее (раздача форм, манера) на bar · крупное (режим, сцена, приём) на phrase — наш QUANT ровно это, дефолт bar подтверждён практикой. Тишина/брейк = покой, экран не трясётся от шума (gate по rmsSlow). Габитуация: одинаковый ответ на каждый кик замыливается — Auto-randomise на такт (§15.1) + вариация глубин уже лечат.

**Книги/статьи (полка BRYK):** Spinrad «The VJ Book» (2005, есть на archive.org) · Faulkner/D-Fuse «VJ: Audio-Visual Art and VJ Culture» (2006) · Eskandar «vE-jA: Art + Technology of Live Audio-Video» (2006) · Jaeger «VJ: Live Cinema Unraveled» (2006) · Gibson/Arisona et al. «Live Visuals: History, Theory, Practice» (Routledge 2022, open-access — проверить при билде) · Bello et al. «A Tutorial on Onset Detection in Music Signals» (2005) · Stowell & Plumbley «Adaptive whitening for real-time onset detection» (2007).

**В код идёт:** CAL noise-floor · band-селектор бита · refractory+whitening в onset · rmsSlow-gate покоя. Трек W3 делегата.
